# Checklist prático de correções

## Objetivo
Transformar a revisão dos docs da Canopi em um plano curto e executável de consolidação, sem reabrir discussões estruturais que já estão suficientemente resolvidas.

## Diagnóstico resumido
A base documental atual já está boa o bastante para orientar produto, arquitetura e páginas.

O foco agora não é refazer a documentação. O foco é reduzir ambiguidade, padronizar linguagem e travar regras operacionais mínimas.

## Ajustes prioritários

### 1. Congelar dicionário oficial de labels
**Por que importa**
Hoje a estrutura conceitual está boa, mas o risco de variação de nomes ainda é alto.

**O que precisa acontecer**
- travar nomes oficiais de páginas
- travar nomes oficiais de entidades
- travar nomes oficiais de camadas do sistema
- travar termos aceitos e termos proibidos

**Status esperado ao concluir**
Qualquer decisão de nomenclatura futura passa a ser validada contra uma referência única.

### 2. Congelar tabela de decisão de fronteiras
**Por que importa**
Ainda existe risco de sobreposição entre páginas próximas, especialmente entre leitura, estratégia e execução.

**O que precisa acontecer**
- definir quando algo pertence a Sinais
- definir quando algo pertence a Ações
- definir quando algo pertence a Contas
- definir quando algo pertence a ABM, ABX e Inteligência Cruzada

**Status esperado ao concluir**
O time deixa de discutir abstrações vagas e passa a decidir por regra.

### 3. Padronizar template mínimo de página
**Por que importa**
Os docs de páginas estão bons, mas em profundidade desigual.

**O que precisa acontecer**
- adotar uma estrutura única mínima para toda página
- garantir que toda página tenha papel, fronteira, inputs, outputs, entidades relacionadas, KPIs e ações derivadas

**Status esperado ao concluir**
Fica muito mais fácil comparar páginas, revisar lacunas e transformar doc em tela.

### 4. Criar documentos de sustentação do projeto
**Por que importa**
A arquitetura do produto está madura, mas ainda falta o pacote que orienta execução contínua do projeto.

**O que precisa acontecer**
- criar README principal
- consolidar roadmap por macroetapas
- registrar regras operacionais do projeto

**Status esperado ao concluir**
O projeto ganha uma camada mais clara de governança prática.

## Ordem recomendada de execução
1. Dicionário oficial de labels
2. Tabela de decisão de fronteiras
3. Template mínimo de página
4. README principal
5. Roadmap consolidado
6. Regras operacionais do projeto

## Critério de saída desta rodada
A rodada de consolidação termina quando:
- não houver ambiguidade de nomenclatura
- não houver disputa de fronteira entre páginas principais
- existir um template padrão para evolução das páginas
- o projeto tiver README, roadmap e regras mínimas registradas

## O que não fazer agora
- não reabrir escopo da V1 sem motivo forte
- não refazer todas as páginas do zero
- não expandir V2 antes de endurecer a base
- não inflar documentação com versões paralelas da mesma coisa

## Resultado esperado
Ao final dessa rodada, a Canopi passa a ter uma base documental suficientemente firme para virar backlog prático, refinamento de interface e construção de telas com menos retrabalho.
